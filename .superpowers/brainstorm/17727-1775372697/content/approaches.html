Extraction Pipeline Redesign: Two Approaches

Optimize for quality + iterative improvement loop

A

Harden Current Architecture

Keep 3-path + synthesis + validation. Fix what's broken:

  • Add response_format: json_schema (strict) everywhere — eliminates all JSON parse errors
  • Upgrade to gpt-4.1 (better instruction following)
  • Use Files API for PDF upload-once
  • Add issue taxonomy to review UI for pattern detection
  • Add cross-run comparison to track improvement

Pros

  • Minimal code changes
  • Preserves independent signal paths
  • Lower risk — incremental fixes

Cons

  • Merge step still loses info / creates conflicts
  • Semantic path rarely adds value over visual
  • 4 LLM calls per cluster = slow iteration cycles
  • Hard to improve prompts when they're spread across 4 stages
B

Redesign: Single Rich Extraction + Validation

Collapse to 2 stages: Extract (all evidence at once) + Validate.

  • Stage 1 (Table): Pure code — CSV parsing, enrichment stats (no LLM)
  • Stage 2 (Extract): One gpt-4.1 call per cluster with ALL context: PDF pages (figures!) + CSV summary + RAG passages. Strict JSON schema output.
  • Stage 3 (Validate): Judge call with same PDF + extraction result. Strict schema.
  • Files API: upload PDF once, reuse file_id across all calls
  • Review UI: issue taxonomy, batch operations, cross-run diffs

Pros

  • Model sees ALL evidence at once — no merge conflicts
  • Figures + text + data in single context = better behavioral descriptions
  • 2 calls/cluster instead of 4 = faster iteration
  • One prompt to tune instead of 4
  • Strict JSON schema = zero parse failures

Cons

  • Bigger rewrite (but pipeline is only ~1200 lines)
  • Loses independent path signals (but merge was causing conflicts anyway)
  • Single point of failure per cluster (mitigated by validation stage)

Recommendation: B

The merge step is the root cause of most quality issues (conflicting directions, lost info). Giving the model all evidence at once with a strict output schema solves both quality and reliability. The 3-path architecture was defensive against bad LLM outputs — strict structured outputs make that unnecessary.

The iteration loop (extract → review → fix prompt → re-run) is also faster with fewer stages. One extraction prompt to tune instead of four.
